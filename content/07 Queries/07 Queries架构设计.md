# 07 Queries 架构设计

> [!abstract] 核心定位
> `07 Queries` 不是对象页，也不是理论页，而是整个项目的 `问题转写层`、`证据中转层` 和 `方法试验层`。  
> 它的职责不是展示结论，而是把 `NotebookLM`、`GraphRAG`、细读和回写动作组织成可复用的查询工作流。

> [!info] 一句话原则
> `07 Queries` 是车间，不是展厅。  
> 这里保存的是问题、提问路径、结构判断、证据压缩和回写去向，而不是最终定稿。

## 一、为什么要单独做 07 Queries

当前 vault 里的对象页、理论页和方法页已经逐渐稳定，但查询层还处在“案例页平铺”的状态。这样做短期能推进写作，长期会出现三个问题：

- 同类问题无法复用，只能重复提问。
- `NotebookLM` 与 `GraphRAG` 的结果没有稳定落点，容易散失。
- 案例页、批处理页、比较页和临时工作台混在一起，后期难以维护。

因此，`07 Queries` 需要从“零散案例集合”升级成一个有内部层级的查询系统。它既服务于 `04 Institutions`、`05 Concepts`、`06 Theory` 和人物/事件页，也反过来把对象层的空白重新转写成下一轮可执行问题。

## 二、07 Queries 在整个项目中的位置

> [!tip] 查询层的上下游
> 上游是 `方法页 + 理论页 + 当前研究目标`。  
> 中层是 `07 Queries`。  
> 下游是 `人物页 / 事件页 / 制度页 / 概念页 / 理论页 / 论文段落`。

可以把整个工作流理解成下面这条链：

`研究意图 -> 问题成形 -> 工具提问 -> 结构判断 -> 例子与印证 -> 回写对象页 -> 再次暴露新问题`

在这条链里：

- `NotebookLM` 负责方法框架、理论锚定、中文解释和长段论证。
- `GraphRAG` 负责英文文本中的结构证据、人物关系、事件链和制度性细节。
- `Obsidian` 负责把查询结果挂接进知识网络。
- `Codex` 负责统一页面骨架、批量回写、格式整理和查询沉淀。

## 三、推荐的内部目录

> [!example] 目录骨架
> `07.1_Templates`
> `07.2_Batch_Tasks`
> `07.3_Case_Studies`
> `07.4_Cross_Matrices`
> `07.5_Workspaces`
> `07.9_Archive`

### 1. `07.1_Templates`

用于保存稳定查询模板，而不是具体案例。

适合放：

- 人物提问模板
- 事件提问模板
- 制度提问模板
- 理论页配套提问模板
- `GraphRAG global` 的固定英文问法

命名前缀建议：

- `QTpl_`

示例：

- `QTpl_叙事黏附与代词消歧测试`
- `QTpl_制度卡四段式提问`
- `QTpl_GraphRAG事件结构英文问法`

### 2. `07.2_Batch_Tasks`

用于批量处理同一类对象，强调规模化、流程化和可重复执行。

适合放：

- 核心人物第二组批处理
- 多个制度卡统一补厚
- 一组事件的结构比较
- 一组概念的定义与误读整理

命名前缀建议：

- `QBatch_`

示例：

- `QBatch_核心人物恩宠与失宠图谱`
- `QBatch_修道院相关制度批处理`

### 3. `07.3_Case_Studies`

用于单个高价值案例，强调完整问题链、证据链和回写动作。

适合放：

- 沃尔西失势
- 国王大事与脱离罗马
- 安妮案
- 修道院解散
- 摩尔受审

命名前缀建议：

- `QCase_`

示例：

- `QCase_《至尊法案》的合法性话语建构`
- `QCase_托马斯·摩尔审讯中的沉默与边界`

### 4. `07.4_Cross_Matrices`

用于横向比较和矩阵型查询，不以单一对象为中心，而以关系和对照为中心。

适合放：

- 同案人物比较
- 多个角色的恩宠曲线对照
- 制度与事件交叉表
- 意象、物质、空间、语言的并置分析

命名前缀建议：

- `QCross_`

示例：

- `QCross_安妮案同案人物比较`
- `QCross_“铁”与“墨水”的物质性意象对峙`

当前已经验证可行的矩阵类型：

- `人物功能矩阵`
  - 例：[[QCross_安妮案同案人物比较]]
- `物质媒介矩阵`
  - 例：[[07 Queries/07.4_Cross_Matrices/QCross_“铁与血”vs“纸与墨”的物质意象对峙]]
- `话语装置矩阵`
  - 例：[[07 Queries/07.4_Cross_Matrices/QCross_誓言、供词、沉默三种话语装置比较]]
- `恩宠机制矩阵`
  - 例：[[07 Queries/07.4_Cross_Matrices/QCross_恩宠、失宠与替代者上位机制]]
- `空间制度矩阵`
  - 例：[[07 Queries/07.4_Cross_Matrices/QCross_宫廷内室、议会与修道院三种权力空间]]
- `概念生产矩阵`
  - 例：[[07 Queries/07.4_Cross_Matrices/QCross_合法性如何在血统、法律与叙述中被制造]]

这说明 `07.4_Cross_Matrices` 不再只是“并排比较几个人”，而已经承担起把人物层、事件层、制度层、概念层和理论层压缩成中层结构判断的功能。

### 5. `07.5_Workspaces`

用于临时试验、探索性查询和工作台，不要求一次成稿，但要求后续可归档或升级。

适合放：

- 还没稳定的问题草稿
- 新工具测试
- 一次性对照表
- 给 `NotebookLM` 或 `GraphRAG` 试问的临时页

命名前缀建议：

- `QWS_`
- `QCanvas_`

示例：

- `QWS_安妮案证词生成试问`
- `QCanvas_沃尔西倒台后的财产与人员流转网络`

### 6. `07.9_Archive`

用于存放已经完成迁移、证据不足、价值较低或被新页面替代的旧查询页。

原则：

- 不轻易删除
- 只归档，不混放
- 在归档前标注去向

## 四、每一类页面的最小骨架

> [!important] 统一模板
> 无论是案例页还是批处理页，最少都应保留“问题界定、提问记录、结构判断、例子与印证、回写去向、残留问题”这六层。

推荐 frontmatter：

```yaml
---
title: 页面标题
note_type: query
status: seed
query_type: case-study
writeback_status: partial
evidence_gap: none
archive_ready: false
target_entities:
  - Cromwell
applied_theory:
  - 新历史主义
tools:
  - NotebookLM
  - GraphRAG
tags:
  - wolfhall
  - query
---
```

推荐正文骨架：

```markdown
## 问题界定

## 适用对象与输入来源

## 提问记录与工具调用

## 结构判断

## 例子与印证

## 回写去向与执行

## 残留问题
```

对于 `QCross` 页，建议在“回写去向与执行”中固定保留一组更明确的复选状态：

- `已回写人物页`
- `已回写事件页`
- `已回写制度页`
- `已回写概念页`
- `已回写理论页`
- `已挂接相关查询页`

不必每张都全部出现，但至少要保留与该页最相关的几项。这样后面看板层和批量筛查时，不再需要重新阅读全文判断进度。

## 五、命名法与状态管理

> [!warning] 不再继续用“页面名各写各的”方式
> 07 层后续要承担批处理和复用功能，所以命名前缀必须稳定。

推荐前缀：

- `QTpl_`：模板
- `QBatch_`：批处理
- `QCase_`：案例
- `QCross_`：比较矩阵
- `QWS_`：临时工作台
- `QCanvas_`：画布型查询

推荐状态：

- `seed`：问题刚成形
- `running`：正在提问和整理
- `stable`：结构已稳定，可反复复用
- `written-back`：已完成主要回写
- `archived`：已归档

推荐辅助字段：

- `writeback_status`：`not-started` / `partial` / `complete`
- `evidence_gap`：当前最主要的证据缺口，用短语写清，例如 `graphrag-close-reading-examples`
- `archive_ready`：`true` / `false`，用于区分“已能冻结”与“仍需追问”

对 `QCross` 而言，这三个字段现在已经可以按更具体的口径理解：

- `writeback_status: complete`
  - 表示主要回写层和相关查询挂接已经完成，不等于证据已经饱和。
- `evidence_gap`
  - 优先写最影响下一轮 `GraphRAG` 或 close reading 的那一个缺口，不保留散乱长清单。
- `archive_ready: true`
  - 只给结构稳定、互链完成、近期不准备继续扩写的一批页。

这一口径尤其适用于 `07.4_Cross_Matrices`，因为矩阵页一旦进入多层回写，最容易出现“内容看起来成熟、状态却无法快速判断”的维护问题。

## 六、07.4 这一层现在已经形成的方法逻辑

> [!success] 从“比较页”升级为“矩阵层”
> 经过这一轮沉淀，`07.4_Cross_Matrices` 已经可以被明确视为 `07 Queries` 内部最重要的中层压缩区。

它当前已经形成的工作逻辑，不再是“把几个对象并排写一下”，而是：

1. 先从 `QCase`、对象页、制度页或理论页中抽出一个真正跨层的问题。
2. 再把这个问题压成一张矩阵，明确每一列比较的不是信息量，而是结构功能。
3. 然后把矩阵页反向回写到 `04 Institutions`、`05 Concepts`、`06 Theory`，必要时回写到人物页和事件页。
4. 最后再把同类 `QCross` 页互相挂接，形成“矩阵之间彼此支撑”的网络，而不是一堆并列孤页。

按现在已经跑通的经验，`QCross` 最适合承担三种任务：

- 把多个个案压成一个更高层的共通机制。
- 把抽象概念拆成几个可观察、可回写的生产路径。
- 为论文段落预先生成可直接使用的比较结构。

也正因为如此，`07.4` 不该继续随手扩页，而应尽量优先新增那些能够同时回写两层以上对象的高压缩矩阵。
