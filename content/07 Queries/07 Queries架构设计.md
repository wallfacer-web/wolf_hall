# 07 Queries 架构设计

> [!abstract] 核心定位
> `07 Queries` 不是对象页，也不是理论页，而是整个项目的 `问题转写层`、`证据中转层` 和 `方法试验层`。  
> 它的职责不是展示结论，而是把 `NotebookLM`、`GraphRAG`、细读和回写动作组织成可复用的查询工作流。

> [!info] 一句话原则
> `07 Queries` 是车间，不是展厅。  
> 这里保存的是问题、提问路径、结构判断、证据压缩和回写去向，而不是最终定稿。

## 一、为什么要单独做 07 Queries

当前 vault 里的对象页、理论页和方法页已经逐渐稳定，但查询层还处在“案例页平铺”的状态。这样做短期能推进写作，长期会出现三个问题：

- 同类问题无法复用，只能重复提问。
- `NotebookLM` 与 `GraphRAG` 的结果没有稳定落点，容易散失。
- 案例页、批处理页、比较页和临时工作台混在一起，后期难以维护。

因此，`07 Queries` 需要从“零散案例集合”升级成一个有内部层级的查询系统。它既服务于 `04 Institutions`、`05 Concepts`、`06 Theory` 和人物/事件页，也反过来把对象层的空白重新转写成下一轮可执行问题。

## 二、07 Queries 在整个项目中的位置

> [!tip] 查询层的上下游
> 上游是 `方法页 + 理论页 + 当前研究目标`。  
> 中层是 `07 Queries`。  
> 下游是 `人物页 / 事件页 / 制度页 / 概念页 / 理论页 / 论文段落`。

可以把整个工作流理解成下面这条链：

`研究意图 -> 问题成形 -> 工具提问 -> 结构判断 -> 例子与印证 -> 回写对象页 -> 再次暴露新问题`

在这条链里：

- `NotebookLM` 负责方法框架、理论锚定、中文解释和长段论证。
- `GraphRAG` 负责英文文本中的结构证据、人物关系、事件链和制度性细节。
- `Obsidian` 负责把查询结果挂接进知识网络。
- `Codex` 负责统一页面骨架、批量回写、格式整理和查询沉淀。

## 三、推荐的内部目录

> [!example] 目录骨架
> `07.1_Templates`
> `07.2_Batch_Tasks`
> `07.3_Case_Studies`
> `07.4_Cross_Matrices`
> `07.5_Workspaces`
> `07.9_Archive`

### 1. `07.1_Templates`

用于保存稳定查询模板，而不是具体案例。

适合放：

- 人物提问模板
- 事件提问模板
- 制度提问模板
- 理论页配套提问模板
- `GraphRAG global` 的固定英文问法

命名前缀建议：

- `QTpl_`

示例：

- `QTpl_叙事黏附与代词消歧测试`
- `QTpl_制度卡四段式提问`
- `QTpl_GraphRAG事件结构英文问法`

### 2. `07.2_Batch_Tasks`

用于批量处理同一类对象，强调规模化、流程化和可重复执行。

适合放：

- 核心人物第二组批处理
- 多个制度卡统一补厚
- 一组事件的结构比较
- 一组概念的定义与误读整理

命名前缀建议：

- `QBatch_`

示例：

- `QBatch_核心人物恩宠与失宠图谱`
- `QBatch_修道院相关制度批处理`

### 3. `07.3_Case_Studies`

用于单个高价值案例，强调完整问题链、证据链和回写动作。

适合放：

- 沃尔西失势
- 国王大事与脱离罗马
- 安妮案
- 修道院解散
- 摩尔受审

命名前缀建议：

- `QCase_`

示例：

- `QCase_《至尊法案》的合法性话语建构`
- `QCase_托马斯·摩尔审讯中的沉默与边界`

### 4. `07.4_Cross_Matrices`

用于横向比较和矩阵型查询，不以单一对象为中心，而以关系和对照为中心。

适合放：

- 同案人物比较
- 多个角色的恩宠曲线对照
- 制度与事件交叉表
- 意象、物质、空间、语言的并置分析

命名前缀建议：

- `QCross_`

示例：

- `QCross_安妮案同案人物比较`
- `QCross_“铁”与“墨水”的物质性意象对峙`

### 5. `07.5_Workspaces`

用于临时试验、探索性查询和工作台，不要求一次成稿，但要求后续可归档或升级。

适合放：

- 还没稳定的问题草稿
- 新工具测试
- 一次性对照表
- 给 `NotebookLM` 或 `GraphRAG` 试问的临时页

命名前缀建议：

- `QWS_`
- `QCanvas_`

示例：

- `QWS_安妮案证词生成试问`
- `QCanvas_沃尔西倒台后的财产与人员流转网络`

### 6. `07.9_Archive`

用于存放已经完成迁移、证据不足、价值较低或被新页面替代的旧查询页。

原则：

- 不轻易删除
- 只归档，不混放
- 在归档前标注去向

## 四、每一类页面的最小骨架

> [!important] 统一模板
> 无论是案例页还是批处理页，最少都应保留“问题界定、提问记录、结构判断、例子与印证、回写去向、残留问题”这六层。

推荐 frontmatter：

```yaml
---
title: 页面标题
note_type: query
status: seed
query_type: case-study
target_entities:
  - Cromwell
applied_theory:
  - 新历史主义
tools:
  - NotebookLM
  - GraphRAG
tags:
  - wolfhall
  - query
---
```

推荐正文骨架：

```markdown
## 问题界定

## 适用对象与输入来源

## 提问记录与工具调用

## 结构判断

## 例子与印证

## 回写去向与执行

## 残留问题
```

## 五、命名法与状态管理

> [!warning] 不再继续用“页面名各写各的”方式
> 07 层后续要承担批处理和复用功能，所以命名前缀必须稳定。

推荐前缀：

- `QTpl_`：模板
- `QBatch_`：批处理
- `QCase_`：案例
- `QCross_`：比较矩阵
- `QWS_`：临时工作台
- `QCanvas_`：画布型查询

推荐状态：

- `seed`：问题刚成形
- `running`：正在提问和整理
- `stable`：结构已稳定，可反复复用
- `written-back`：已完成主要回写
- `archived`：已归档

也就是说，`07 Queries` 不是额外层，而是把“方法”变成“生产”的中间层。
