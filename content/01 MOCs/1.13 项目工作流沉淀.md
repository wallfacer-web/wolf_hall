# 项目工作流沉淀

这页只沉淀已经跑通、明天可以直接复用的工作方式，避免重启后重新摸索。

## 一句话总原则

- `NotebookLM` 负责来源扎根、方法框架、制度解释与长段论证材料。
- `GraphRAG` 负责英文文本检索、结构判断、人物卡与事件卡补厚。
- `Obsidian` 负责把对象页、方法页、总览页、附录页挂成可持续知识网络。
- `Codex` 负责统一骨架、批量回写、结构维护、入口整理与沉淀。

## GraphRAG 的固定查询方式

以后不要改命令形式，事件层和对象层都优先复用这套：

```powershell
conda activate Graph
cd H:\graphrag-0.5.0\ragtest
python query_ollama.py --root . --method global "English question"
```

补充规则：

- 当前最稳的是 `global`
- `local` 暂时不稳，不作为默认工作流
- 问题尽量用英文，因为 trilogy corpus 是英文文本索引
- 当前索引目录以 `H:\graphrag-0.5.0\ragtest` 这一套工作方式为准

## NotebookLM 的固定用途

适合问这些问题：

- 数字人文方法如何研究人物、事件、制度、时空、情感
- 某个制度或背景知识如何向中文读者解释
- 某一事件或人物的意义层、证据层、长段论证如何组织
- 如何区分“来源扎根总结”和“面向《狼厅》的应用推演”

不适合让它直接替代对象页回写。对象页回写仍以结构压缩后的笔记为主。

## 已经沉淀出的稳定对象流程

### 做一个人物

1. 从 [[1.01 人物总览|人物总览]] 找到目标人物。
2. 用 [[1.08 人物分析方法|人物分析方法]] 确定人物要压成什么结构角色。
3. 用英文问题跑 `GraphRAG global`。
4. 回写人物卡，保留丰富关系链、人物气质、结构角色、论文判断。
5. 必要时把人物挂回比较页或案例页。

### 做一个事件

1. 从 [[1.02 事件总览|事件总览]] 确认事件在时间链中的位置。
2. 用 [[1.09 事件分析方法|事件分析方法]] 确认要补的栏目。
3. 用 [[1.11 事件研究提问指南|事件研究提问指南]] 的三问模板跑英文 `global`。
4. 先写结构判断，再用 `NotebookLM` 补意义层与证据层。
5. 把成熟事件挂回总览、双向索引或案例页。

### 做一个制度

1. 先确认它是否真的是“制度”，而不是更适合放进 `05 Concepts` 的概念。
2. 制度卡统一写这几层：
   - 制度定义
   - 制度结构
   - 在《狼厅》中的作用
   - 中文读者最容易误解之处
   - 相关事件
   - 相关人物
   - 可直接用于论文的压缩判断
3. 对中文读者理解门槛高的制度，先向 `NotebookLM` 问“四段式制度卡框架”。
